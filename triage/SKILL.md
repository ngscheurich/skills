# Triage

Move **briefs** in `briefs/` through a small state machine of triage tags. See `CONTEXT.md` for the ubiquitous language; the triage-as-tags decision is recorded in the v1 PRD's resolved decisions.

Every triage-notes section appended to a brief during this skill **must** start with this disclaimer:

```
> *These notes were generated by AI during triage.*
```

## What triage touches (and what it doesn't)

This skill only modifies a brief's `tags:` frontmatter list and, when needed, its body (appending a `## Triage Notes` section). It may also suggest a change to `type:` if a brief is misclassified. It writes `status:` in just two sanctioned cases — setting `closed` when applying `wontfix` (see below), and promoting a `draft` to `ready` when the developer explicitly **accepts** it (see below) — and otherwise **never** touches it; the system owns every other status write per ADR-0008.

Note the `status:` axis is **orthogonal** to triage tags. A brief can be `status: draft` (captured, not yet committed to land) while carrying any disposition tag — including `agent-ready`. Assigning `agent-ready` / `human-only` says the brief is *workable*; it does **not** by itself accept the brief. Acceptance (`draft → ready`) is a separate, deliberate decision. Triage may *recommend* "this draft looks ready to accept," but only writes `draft → ready` when the developer says so.

## Roles

**Type** (lives in the brief's `type:` frontmatter — set at authoring time; triage may suggest changes):

- `feature` — new capability
- `bug` — something is broken
- `refactor` — restructure without changing behavior
- `chore` — maintenance
- `docs` — documentation only

**Triage tags** (live in the brief's `tags:` frontmatter list — an open list; a triaged brief carries one of these as its disposition, possibly alongside other tags):

- `needs-triage` — needs evaluation
- `needs-info` — waiting on the requester for more information
- `agent-ready` — fully specified, an agent can grab it in a workspace with no further human context
- `human-only` — requires a human (judgment call, external access, design decision, manual testing)
- `wontfix` — will not be actioned

A brief that arrives without any of these tags is treated as `needs-triage`. Other tags may coexist with the disposition, but if two of these *disposition* tags conflict, flag it and ask the user before doing anything else.

**State transitions:** an untriaged brief normally moves to `needs-triage` first; from there it moves to `needs-info`, `agent-ready`, `human-only`, or `wontfix`. `needs-info` returns to `needs-triage` once the requester replies. The user can override at any time — flag transitions that look unusual and ask before proceeding.

## Invocation

The user invokes `/triage` and describes what they want in natural language. Interpret the request and act. Examples:

- "Show me anything that needs my attention"
- "Let's look at `briefs/foo-bar.md`" / "Let's look at `[[foo-bar]]`"
- "Move `[[foo-bar]]` to agent-ready"
- "What's ready for agents to pick up?"

## Show what needs attention

Scan `briefs/*.md`, parse each file's frontmatter, and present three buckets, oldest first (file mtime as proxy for age):

1. **untagged** — brief has no triage tag in `tags:`. Never triaged.
2. **`needs-triage`** — evaluation in progress.
3. **`needs-info` with body changes since the last triage notes** — needs re-evaluation (the requester has likely replied).

Show counts and a one-line summary per brief (use the title — first H1 of the body). Let the user pick.

## Triage a specific brief

1. **Gather context.** Read the full brief body and frontmatter. Parse any prior `## Triage Notes` section so you don't re-ask resolved questions. Explore the codebase using the project's ubiquitous language from `CONTEXT.md`, respecting ADRs in the area.

2. **Recommend.** Tell the user your recommended triage tag (and, if `type:` looks wrong, a proposed change) with reasoning, plus a brief codebase summary relevant to the brief. Wait for direction.

3. **Reproduce (bugs only).** Before any grilling, attempt reproduction: read the requester's steps, trace the relevant code, run tests or commands. Report what happened — successful repro with code path, failed repro, or insufficient detail (a strong `needs-info` signal). A confirmed repro makes a much sharper brief.

4. **Grill (if needed).** If the brief needs fleshing out, run a `/grill-with-docs` session.

5. **Apply the outcome:**
   - `agent-ready` — verify the brief body is sharp enough for an agent to work from (acceptance criteria, what to build, no stale file paths). If it isn't, sharpen it inline per the template in `/to-briefs`. Then write the `agent-ready` tag.
   - `human-only` — sharpen the body the same way, then write the `human-only` tag and append a `## Triage Notes` section noting *why* it can't be delegated (judgment calls, external access, design decisions, manual testing).
   - `needs-info` — append a `## Triage Notes` section using the template below. Write the `needs-info` tag.
   - `wontfix` — append a `## Triage Notes` section explaining why it will not be actioned. Write the `wontfix` tag and set `status: closed` — the one status this skill writes, because `closed` is terminal and no workspace transition produces it (ADR-0008). Exception: if the work actually landed elsewhere (absorbed into another slice), set `status: done` instead, since the change is in the codebase.
   - `needs-triage` — write the tag. Optional `## Triage Notes` if there's partial progress.
   - **accept** (`draft → ready`) — when the developer explicitly commits to landing a `draft` brief, set `status: ready`. This is orthogonal to the disposition tag (an accepted brief still needs to be `agent-ready` or `human-only` to be worked) — only do it on an explicit acceptance, never as a side-effect of assigning a disposition.

## Quick tag override

If the user says "move `[[foo-bar]]` to agent-ready", trust them and apply the tag directly. Confirm what you're about to do (tag change, any body edits), then act. Skip grilling. If moving to `agent-ready` without a grilling session, ask whether they want to sharpen the body first.

## Editing the brief safely

- Always preserve the brief's first H1 (the canonical title).
- `tags:` is a flat YAML list. Add/remove individual entries; don't replace the whole list unnecessarily.
- Do not touch `status:` except to set `closed` alongside the `wontfix` tag (or `done` when work was absorbed), or to promote `draft → ready` on an explicit developer acceptance — the system owns every other status write (ADR-0008).
- The `## Triage Notes` section, if present, lives at the bottom of the body. Append new notes; don't overwrite prior ones.

## Triage-notes template

Append (or update) this section at the bottom of the brief body:

```markdown
## Triage Notes

> *These notes were generated by AI during triage.*

**What we've established so far:**

- point 1
- point 2

**What we still need (from the requester):**

- question 1
- question 2
```

Capture everything resolved during grilling under "established so far" so the work isn't lost. Questions must be specific and actionable, not "please provide more info".

## Resuming a previous session

If prior `## Triage Notes` exist on the brief, read them, check whether the body has changed since (the requester may have edited the brief to answer outstanding questions), and present an updated picture before continuing. Don't re-ask resolved questions.
